## Context

当前项目 Brevis Vera 是一个数字媒体真实性认证系统，使用 C2PA 标准进行来源验证，并支持图像编辑功能。ZK 证明模块目前只是一个占位符实现，生成的只是模拟数据而非真正的零知识证明。

Pico ZKVM 是一个轻量级的零知识证明虚拟机，专为资源受限环境设计，可以生成简洁的证明文件。它能够证明某个计算的输出是正确的。

## Goals / Non-Goals

**Goals:**
- 集成 Pico ZKVM 作为 ZK 证明生成的后端
- 实现图像编辑转换的 ZK 电路，支持 crop、resize、brightness 三种转换
- 提供独立的证明验证功能，无需访问原始图像
- 保持证明文件体积小巧，适合网络传输

**Non-Goals:**
- 支持视频文件的 ZK 证明（不在本次范围内）
- 实现递归零知识证明
- 支持多方计算的 ZK 证明
- 实现实时的流式证明生成

## Decisions

### 1. 使用 Pico ZKVM 而非其他 ZK 框架

**决定**: 采用 Pico ZKVM 作为 ZK 证明引擎

**理由**:
- Pico ZKVM 专为嵌入式和轻量级场景设计，证明文件体积小
- 与 Rust 生态系统集成良好
- 生成的证明可以独立验证，无需信任证明生成者

**替代方案考虑**:
- Groth16: 需要可信设置流程，证明较大
- STARKs: 不需要可信设置，但证明较大，验证较慢
- Bulletproofs: 适合范围证明，但不适合通用计算

### 2. 电路设计策略

**决定**: 电路输入包括原图像，编辑后的图像，声明的转换操作。通过在zkvm电路里对原图像进行声明的转换操作后，计算图像的hash与输入的编辑后的图像的hash相同。除了生成proof之外，commit的output需要包含原图像的hash，编辑后的图像hash。

### 3. 证明格式

**决定**: 使用 JSON 格式存储证明和验证密钥

**理由**:
- 易于调试和查看
- 与现有系统兼容
- 可被其他语言轻松解析

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Pico ZKVM 性能问题 | 大图像处理慢 | 先对图像做哈希，电路基于哈希工作 |
| 证明大小 | 网络传输成本 | Pico ZKVM 本身已优化 |
| 兼容性 | 依赖更新可能导致破坏 | 锁定版本，定期更新 |
| 安全性 | ZK 电路可能存在漏洞 | 审计电路代码，添加测试 |
